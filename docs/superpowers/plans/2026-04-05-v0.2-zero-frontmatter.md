# sdcoh v0.2 — Zero-frontmatter mode (rules-based dependencies)

## Goal

依存関係をすべて `sdcoh.yml` のルールで宣言する。各Markdownファイルから `sdcoh:` frontmatterを不要にする。

**Why:** 新エピソード追加のたびにfrontmatter編集するのが煩雑。設計書の増減でfrontmatter書き換えが発生する。設定一本化で運用コストゼロにする。

**Breaking change:** v0.1.x のfrontmatter mode は削除。backward-compat shim は入れない (YAGNI)。

---

## 新 `sdcoh.yml` の形

```yaml
project:
  name: "贖罪の壊し方"
  alias: "headhunter"

# ディレクトリ → node_type マッピング
scan:
  - { path: "research/",  type: "research" }
  - { path: "design/",    type: "design" }
  - { path: "briefs/",    type: "brief" }
  - { path: "revision3/", type: "episode" }
  - { path: "reviews/",   type: "review" }

node_types:
  research: { layer: -1 }
  design:   { layer: 0 }
  brief:    { layer: 1 }
  episode:  { layer: 2 }
  review:   { layer: 3 }

# 依存ルール: from が変わると to が陳腐化する
rules:
  - name: "design informs episodes"
    from: "design/*.md"
    to:   "revision3/ep*.md"
    relation: informs

  - name: "brief feeds episode"
    from: "briefs/{ep}-*.md"
    to:   "revision3/{ep}.md"
    relation: feeds

  - name: "episode reviewed"
    from: "revision3/{ep}.md"
    to:   "reviews/{ep}-*.md"
    relation: reviewed_by
```

---

## セマンティクス統一

**エッジの向き: `from → to` = 「fromが変わるとtoが陳腐化する」**

旧v0.1では `depends_on` と `updates` の2方向があったが、v0.2では1方向に統一。意味論的にすべて「upstream→downstream」のchange propagationとして扱う。

- `status` 判定: `from.mtime > to.mtime` → to が stale
- `impact` 判定: あるノード変更の影響先 = そのノードが `from` になってるルールの `to` 側を再帰的に辿る

## パターン構文

### from / to パターン

| 構文 | 意味 | 例 |
|------|------|-----|
| `*` | スラッシュ以外の任意文字列 | `design/*.md` |
| `{name}` | キャプチャ変数 (非貪欲, スラッシュ不可) | `briefs/{ep}-*.md` |
| リテラル | そのままマッチ | `revision3/ep01.md` |

### 正規表現への変換ルール

- `{name}` → `(?P<name>[^/]+?)`
- `*` → `[^/]*`
- `?` → `[^/]`
- `.` → `\.`
- その他の正規表現メタ文字 → エスケープ

### 変数の展開

1. `from` パターンから全キャプチャを抽出
2. `to` パターンの `{name}` をキャプチャ値で置換
3. 置換後パターン（`*` を含みうる）をファイルパスとグロブマッチ
4. マッチしたファイルそれぞれにエッジを作る

### Node ID 生成

`{type}:{basename}` (拡張子除く)

例:
- `design/characters.md` → `design:characters`
- `briefs/ep01-kubota-brief.md` → `brief:ep01-kubota-brief`
- `revision3/ep11.md` → `episode:ep11`

## 想定される落とし穴

1. **非貪欲マッチの挙動**: `{ep}` が `ep11-kubota-brief` 全体ではなく `ep11` だけをキャプチャするよう、後続パターンとの境界を明確に。テストで固める。

2. **自己ループ**: 同一ファイルが from と to 両方にマッチするケース。source path == target path なら edge を作らない。

3. **fan-out の爆発**: `design/*.md → revision3/ep*.md` が N×M エッジを作る。headhunter なら 5設計書 × 18話 = 90エッジ。許容範囲。

4. **ルールのマッチ0件**: タイポや未作成ディレクトリ検出のため warning は出す。

5. **Node ID衝突**: `rglob("*.md")` で再帰走査するため、同じ scan entry 内のサブディレクトリに同名ファイルがあると `{type}:{basename}` が重複する。**検出したら error で止める**（§2 の検証項目）。

---

## 代替案（却下）

### A案: frontmatter と rules の併用
- 却下理由: YAGNI。運用がハイブリッドになると混乱する。

### B案: レイヤーから自動的に全 upstream→downstream エッジを張る
- 却下理由: エッジが多すぎる（5層なら組み合わせ爆発）。意味のない edge がノイズになる。明示的ルールが正解。

### C案: Python DSLで書く
- 却下理由: YAML1本のほうが読みやすい。スキルから編集もしやすい。
